================================================================================
  TITANBAY API — PHASE 2 WALKTHROUGH
  Architecture upgrades: DI, Repository pattern, Result pattern,
  Domain errors, Caching, Logging, and Decorator pattern.
  Written for interview preparation.
================================================================================


================================================================================
  WHAT CHANGED FROM PHASE 1 — THE BIG PICTURE
================================================================================

Phase 1 had a working API with a simple flat structure:
  Route → Controller → Service (with inline DB queries) → Response

Phase 2 adds several real-world engineering patterns that you'd see in a
production TypeScript codebase:

  1. Dependency Injection (tsyringe)     — loose coupling, testability
  2. Repository Pattern                  — abstracted data access layer
  3. Result<T, E> Pattern               — explicit error handling without throws
  4. Domain Error Hierarchy              — typed, meaningful errors
  5. In-Memory Caching (node-cache)      — performance optimisation
  6. Structured Logging (pino)           — production observability
  7. Decorator Pattern for services      — separation of concerns (SRP)
  8. Centralised constants               — no magic strings scattered in code
  9. Pagination                          — all list endpoints paginated
 10. DTOs                                — clean separation of DB models and API responses

The architecture went from:
  Controller → Service → Prisma

To:
  Controller → [Caching Decorator → Logging Decorator → Core Service] → Repository → Prisma

Each layer has ONE job and nothing else.


================================================================================
  1. DEPENDENCY INJECTION (tsyringe)
================================================================================

WHAT IS DI?
-----------
Dependency Injection means: instead of a class creating its own dependencies,
they are "injected" from outside. The class declares what it needs; something
else provides it.

In C# terms: this is exactly like ASP.NET Core's built-in DI container.
  services.AddScoped<IFundService, FundService>();  ← C#
  container.register(TOKENS.FundService, { useClass: FundService });  ← our code

WHY DOES IT MATTER?
-------------------
Without DI:
  class FundController {
    private service = new FundService(new PrismaClient());  // ← hardcoded
  }
  // To test FundController you MUST create a real FundService with a real DB

With DI:
  class FundController {
    constructor(@inject(TOKENS.FundService) private service: IFundService) {}
  }
  // To test FundController, inject a mock IFundService — no DB needed

WHAT WE USED: tsyringe
-----------------------
tsyringe is a lightweight DI container for TypeScript (from Microsoft).
It uses decorators:

  @injectable()       — marks a class as resolvable by the container
  @inject(TOKEN)      — specifies which token to use for constructor injection

Setup requirements:
  - reflect-metadata must be imported FIRST (before any tsyringe code)
  - tsconfig needs: "experimentalDecorators": true, "emitDecoratorMetadata": true
  - These settings tell TypeScript to emit type metadata so tsyringe can
    read constructor parameter types at runtime

HOW TOKENS WORK
---------------
A "token" is how you label something in the DI container.
We use Symbol.for() which creates a globally unique symbol from a string:

  // src/constants/tokens.ts
  export const TOKENS = {
    FundService: Symbol.for('IFundService'),
    FundRepo:    Symbol.for('IFundRepository'),
    Logger:      Symbol.for('ILogger'),
    // ...
  } as const;

Symbol.for('IFundService') always returns the same symbol for the same string,
even across different module imports. This prevents subtle bugs.

We keep ALL tokens in ONE file so:
  - One place to audit all registered things
  - Renaming a service = update one entry
  - Autocomplete works: TOKENS.FundService is safer than 'IFundService' string

THE CONTAINER (src/container.ts)
---------------------------------
All DI wiring lives in one file. Controllers, services, repositories, and
infrastructure are all registered here:

  export function configureContainer() {
    // Infrastructure singletons
    container.registerInstance(TOKENS.PrismaClient, prisma);
    container.registerSingleton(TOKENS.CacheService, NodeCacheService);
    container.registerSingleton(TOKENS.Logger, PinoLogger);

    // Repositories
    container.registerSingleton(TOKENS.FundRepo, PrismaFundRepository);

    // Core service (pure business logic)
    container.registerSingleton(TOKENS.FundServiceCore, FundService);

    // Decorated service (what controllers actually get)
    container.register(TOKENS.FundService, {
      useFactory: (c) => {
        const core = c.resolve(TOKENS.FundServiceCore);
        const cache = c.resolve(TOKENS.CacheService);
        const logger = c.resolve(TOKENS.Logger);
        // Stack: Caching wraps Logging wraps Core
        return new CachingFundService(new LoggingFundService(core, logger), cache);
      }
    });
  }

configureContainer() is called once at startup (in index.ts) before the
server starts accepting requests.

IMPORTANT: the routes use LAZY resolution — they call container.resolve()
inside each request handler, not at startup:

  router.get('/', (req, res, next) =>
    container.resolve(FundController).listFunds(req, res, next)
  );

This means the container doesn't need to be configured at module import time —
it just needs to be ready before the first HTTP request arrives.


================================================================================
  2. REPOSITORY PATTERN
================================================================================

WHAT IT IS
----------
The Repository pattern puts a dedicated class between the service and the
database. The service talks to an INTERFACE; the actual Prisma implementation
is hidden behind it.

  IFundRepository (interface)  ← what the service knows about
       ↑
  PrismaFundRepository         ← the actual Prisma queries

C# comparison: this is exactly Repository<T> or IFundRepository in C# DDD.

WHY BOTHER?
-----------
1. You can swap the DB implementation without changing any service code.
   Want to switch from Postgres to MongoDB? Write a MongoFundRepository,
   register it in the container. Services are untouched.

2. Services become testable without a real database.
   In unit tests, inject a MockFundRepository that returns hardcoded data.

3. Prisma-specific logic (query structure, error codes) stays in one layer.
   Services don't need to know about Prisma.

THE INTERFACE
-------------
  // repositories/interfaces/fund.repository.interface.ts
  export interface IFundRepository {
    findAll(params: PaginationParams): Promise<{ data: Fund[]; total: number }>;
    findById(id: string): Promise<Fund | null>;
    exists(id: string): Promise<boolean>;
    create(data: CreateFundInput): Promise<Fund>;
    update(id: string, data: Omit<UpdateFundInput, 'id'>): Promise<Fund>;
  }

Note: update() takes Omit<UpdateFundInput, 'id'> — TypeScript enforces that
you can NEVER accidentally pass the id inside the data to update. This
catches a real bug we identified: the original code passed the full DTO
including 'id' to the update call.

THE IMPLEMENTATION
------------------
  // repositories/prisma/fund.repository.ts
  @injectable()
  export class PrismaFundRepository implements IFundRepository {
    constructor(@inject(TOKENS.PrismaClient) private readonly prisma: PrismaClient) {}

    async findAll(params: PaginationParams) {
      const [data, total] = await this.prisma.$transaction([
        this.prisma.fund.findMany({ skip, take: limit, orderBy: { created_at: 'desc' } }),
        this.prisma.fund.count(),
      ]);
      return { data, total };
    }
    // ...
  }

$transaction([...]) runs both queries atomically and in one round-trip —
more efficient than two separate queries.

exists() uses COUNT instead of SELECT to check existence — cheaper DB query
than fetching the full record just to see if it's there.


================================================================================
  3. DOMAIN ERROR HIERARCHY
================================================================================

WHAT'S WRONG WITH GENERIC ERRORS?
-----------------------------------
Phase 1 had an AppError class that held a statusCode and a code string.
This works, but it's all in one class — hard to distinguish in catch blocks,
and nothing enforces that error codes match status codes.

THE HIERARCHY
--------------
  DomainError (abstract base)
    ├── NotFoundError (404)
    │     ├── FundNotFoundError
    │     └── InvestorNotFoundError
    ├── DuplicateEmailError (409 CONFLICT)
    ├── ValidationError (400 with field-level details)
    └── InfrastructureError (500 — wraps unexpected DB/external errors)

Each class owns its statusCode and code as readonly properties:

  export class FundNotFoundError extends NotFoundError {
    constructor(id: string) {
      super('Fund', id);
      // Message is auto-generated: "Fund with id '...' was not found"
    }
  }

Benefits:
  - instanceof checks: if (err instanceof NotFoundError) → handles all 404s
  - Consistent messages: all 404s have the same structure
  - statusCode is embedded: the error knows its own HTTP status
  - Grep-friendly: search "FundNotFoundError" to find all places it's thrown


================================================================================
  4. RESULT<T, E> PATTERN
================================================================================

WHAT'S WRONG WITH THROWING EXCEPTIONS?
----------------------------------------
In Phase 1, services threw errors and controllers caught them with try/catch.
Problems:
  - TypeScript can't tell you WHAT errors a function might throw
  - You can forget try/catch and silently miss errors
  - The function signature doesn't tell you about error cases

THE RESULT PATTERN
------------------
Result<T, E> is a container that holds EITHER a success value OR an error.
The caller MUST handle both cases. TypeScript enforces this.

  // lib/result.ts
  export class Result<T, E extends DomainError = DomainError> {
    static ok<T>(value: T): Result<T, never>       // success
    static fail<E>(error: E): Result<never, E>      // failure

    get isOk(): boolean
    get isErr(): boolean
    get value(): T    // throws if isErr
    get error(): E    // throws if isOk
  }

USAGE IN A SERVICE
------------------
  async findById(id: string): Promise<Result<FundResponseDto, DomainError>> {
    const fund = await this.fundRepo.findById(id);
    if (!fund) return Result.fail(new FundNotFoundError(id));
    return Result.ok(toFundResponse(fund));
  }

The return type TELLS you this can fail. You can't ignore it.

USAGE IN A CONTROLLER
---------------------
  const result = await this.service.findById(id);

  if (result.isErr) {
    return res.status(result.error.statusCode).json({
      error: { code: result.error.code, message: result.error.message }
    });
  }

  return res.json(result.value);

This is explicit pattern matching — the same concept as:
  - C#: Result<T> from FluentResults
  - F#/Haskell: Either/Option types
  - Rust: Result<T, E>

WHY NOT JUST USE EXCEPTIONS?
-----------------------------
For truly unexpected errors (DB connection failure, null pointer) — use
exceptions, because you can't recover anyway.

For EXPECTED failure cases (fund not found, duplicate email) — use Result.
These are part of your domain, not exceptional circumstances.


================================================================================
  5. IN-MEMORY CACHING (node-cache)
================================================================================

WHAT AND WHY
------------
The API serves reads far more than writes. Fetching a list of funds from the
DB on every request wastes time and DB resources. An in-memory cache stores
recent results and serves them instantly.

node-cache is a simple in-memory store — like a Map with automatic TTL
(Time To Live) expiry.

THE CACHE SERVICE INTERFACE
-----------------------------
  export interface ICacheService {
    get<T>(key: string): T | undefined;
    set<T>(key: string, value: T, ttl?: number): void;
    del(key: string): void;
    invalidateByPrefix(prefix: string): void;  // ← bulk delete by key prefix
    flush(): void;
  }

The interface means:
  - You can swap node-cache for Redis without changing service code
  - You can inject a NoOpCacheService in tests for predictable behaviour

CACHE KEYS — NO MAGIC STRINGS
-------------------------------
  // constants/cache.ts
  export const CacheKeys = {
    FUNDS_LIST_PREFIX: 'funds:list',
    FUNDS_LIST: (page, limit) => `${CacheKeys.FUNDS_LIST_PREFIX}:p${page}:l${limit}`,
    FUND_BY_ID: (id) => `funds:${id}`,
    // ...
  };

The prefix and the full key are derived from the SAME constant.
Invalidating by prefix uses the same string as building the key:

  this.cache.invalidateByPrefix(CacheKeys.FUNDS_LIST_PREFIX);
  // Deletes ALL cached pages of the fund list

If someone changes the prefix, both the key builder and the invalidation
update together — they reference the same constant. This prevents a subtle bug
where cache invalidation silently stops working.

CACHE-ASIDE PATTERN
--------------------
  1. Check cache → hit? Return immediately.
  2. Miss? Query the database.
  3. Store result in cache.
  4. Return result.

On WRITE operations (create/update):
  - Set the by-ID cache to the new value (cache warming)
  - Invalidate ALL list caches (the new item changes list results)

Cache errors NEVER fail a request. If node-cache crashes, we fall through
to the DB. Cache is an optimisation, not a dependency.

CACHE STAMPEDE NOTE (mentioned but not implemented — good to discuss)
-----------------------------------------------------------------------
If 50 concurrent requests all miss the same cache key simultaneously, all
50 hit the DB. This "thundering herd" problem is solved by Single-Flight
deduplication (lib/single-flight.ts): only ONE DB call is made; all other
callers wait for its result. We built the code but noted it's a stretch
goal — worth mentioning in the interview as "here's what I'd add next".


================================================================================
  6. STRUCTURED LOGGING (pino)
================================================================================

WHAT IS STRUCTURED LOGGING?
-----------------------------
Instead of: console.log("Fund created: " + fund.id)
We use:     logger.info("Fund created", { id: fund.id, duration: "12ms" })

The second form outputs JSON:
  {"level":30,"time":1234567890,"msg":"Fund created","id":"abc-123","duration":"12ms"}

JSON logs can be parsed, filtered, and analysed by tools like Datadog,
Grafana, or AWS CloudWatch. console.log strings cannot.

pino is the fastest Node.js logger — benchmarked at ~6x faster than Winston.

THE LOGGER INTERFACE
---------------------
  export interface ILogger {
    info(message: string, data?: Record<string, unknown>): void;
    warn(message: string, data?: Record<string, unknown>): void;
    error(message: string, data?: Record<string, unknown>): void;
    debug(message: string, data?: Record<string, unknown>): void;
  }

The interface means you can inject a silent NoOpLogger in tests to stop
log noise, or a MockLogger to assert specific logs were made.

LOG MESSAGE CONSTANTS
----------------------
  // constants/log-messages.ts
  export const LOG_MESSAGES = {
    FUND_CREATED: 'Fund created',
    FUND_CREATE_FAILED: 'Fund creation failed',
    CACHE_HIT: 'Cache hit',
    SLOW_QUERY_DETECTED: 'Slow query detected',
    // ...
  } as const;

No magic strings scattered in the codebase. Every log message is searchable.
If you want to find all places "Fund created" is logged — grep LOG_MESSAGES.FUND_CREATED.


================================================================================
  7. DECORATOR PATTERN FOR SERVICES
================================================================================

THE PROBLEM WITH "DOING EVERYTHING IN THE SERVICE"
----------------------------------------------------
A naive FundService might:
  1. Check cache
  2. Log cache hit/miss
  3. Query the database
  4. Handle business rules
  5. Map to DTOs
  6. Warm cache
  7. Log results
  8. Track timing

This violates Single Responsibility Principle — the service has many jobs.
It's also hard to test each concern in isolation.

THE SOLUTION: THREE LAYERS
---------------------------
We split one service into three classes, each implementing IFundService:

  FundService (core)           — ONLY business logic
  LoggingFundService           — ONLY logging
  CachingFundService           — ONLY caching

Each decorator wraps the next one, all implementing the same interface:

  interface IFundService {
    findAll(params): Promise<Result<...>>
    findById(id): Promise<Result<...>>
    create(data): Promise<Result<...>>
    update(data): Promise<Result<...>>
  }

CORE SERVICE — PURE BUSINESS LOGIC
------------------------------------
  export class FundService implements IFundService {
    constructor(@inject(TOKENS.FundRepo) private readonly fundRepo: IFundRepository) {}

    async findAll(params) {
      try {
        const { data, total } = await this.fundRepo.findAll(params);
        return Result.ok(buildPaginatedResponse(toFundResponseList(data), total, params));
      } catch (err) {
        return Result.fail(new InfrastructureError('Failed to fetch funds', err));
      }
    }
  }

No cache code. No logging. Just: call repo → map to DTO → return Result.
This is trivially unit testable with a mock repo.

LOGGING DECORATOR
------------------
  export class LoggingFundService implements IFundService {
    constructor(private inner: IFundService, private logger: ILogger) {}

    async findAll(params) {
      const start = Date.now();
      const result = await this.inner.findAll(params);  // ← delegates to core
      const duration = Date.now() - start;

      if (result.isOk) {
        this.logger.info('Fund list fetched', { count: result.value.data.length, duration });
      } else {
        this.logger.error('Fund list failed', { error: result.error.message });
      }

      if (duration > 200) {
        this.logger.warn('Slow query detected', { operation: 'fund.findAll', duration });
      }

      return result;  // ← passes result through unchanged
    }
  }

It calls the inner service, logs the outcome, then returns the SAME result.
Transparent to the caller — they don't know they're talking to a decorator.

CACHING DECORATOR
------------------
  export class CachingFundService implements IFundService {
    constructor(private inner: IFundService, private cache: ICacheService) {}

    async findAll(params) {
      const key = CacheKeys.FUNDS_LIST(params.page, params.limit);
      const cached = this.cache.get(key);
      if (cached) return Result.ok(cached);  // ← short-circuit

      const result = await this.inner.findAll(params);  // ← cache miss: hit inner
      if (result.isOk) this.cache.set(key, result.value);
      return result;
    }
  }

COMPOSITION IN THE CONTAINER
------------------------------
  container.register(TOKENS.FundService, {
    useFactory: (c) => {
      const core   = c.resolve(TOKENS.FundServiceCore);  // FundService
      const cache  = c.resolve(TOKENS.CacheService);
      const logger = c.resolve(TOKENS.Logger);

      // Stack: Controller → Caching → Logging → Core
      return new CachingFundService(
        new LoggingFundService(core, logger),
        cache
      );
    }
  });

The controller receives the fully-decorated service but has no idea it's
talking to three layers. It just calls service.findAll() and gets a Result.

C# EQUIVALENT
--------------
This is exactly the Decorator pattern with Scrutor in C#:

  services.AddScoped<IFundService, FundService>();
  services.Decorate<IFundService, LoggingFundService>();
  services.Decorate<IFundService, CachingFundService>();

tsyringe's useFactory achieves the same composition manually.

WHY THIS IS BETTER
-------------------
1. Core service is pure — easy to read and unit test
2. Add a new concern (rate limiting, metrics) → add a new decorator
3. Remove caching in tests → register without CachingFundService
4. Each decorator is independently testable
5. Open/Closed principle: add behaviour without modifying existing code


================================================================================
  8. DTOs (DATA TRANSFER OBJECTS)
================================================================================

WHY DTOs?
----------
Prisma returns raw DB models: Fund, Investor, Investment.
These have Decimal objects for money, Date objects for timestamps,
and may contain more fields than the API should expose.

A DTO is a plain object shaped exactly for the API response:

  // The Prisma model (internal)
  Fund { id, name, target_size_usd: Decimal, created_at: Date, ... }

  // The DTO (what the API sends)
  FundResponseDto { id, name, target_size_usd: number, created_at: string, ... }

The mapping functions handle the conversions:

  export function toFundResponse(fund: Fund): FundResponseDto {
    return {
      id: fund.id,
      name: fund.name,
      target_size_usd: fund.target_size_usd.toNumber(),  // Decimal → number
      created_at: fund.created_at.toISOString(),           // Date → string
      status: fund.status,
    };
  }

Benefits:
  - DB schema changes don't automatically break the API contract
  - The type-checker enforces the mapping — missing fields are compile errors
  - Serialisation logic (Decimal.toNumber(), date formatting) is centralised


================================================================================
  9. PAGINATION
================================================================================

All list endpoints now return paginated results:

  GET /funds?page=1&limit=20
  →
  {
    "data": [...],
    "total": 47,
    "page": 1,
    "limit": 20,
    "totalPages": 3
  }

The pagination query is validated with Zod:

  const paginationSchema = z.object({
    page:  z.coerce.number().int().min(1).default(1),
    limit: z.coerce.number().int().min(1).max(100).default(20),
  });

z.coerce converts query string "1" (string) to 1 (number) automatically.
Invalid values (page=0, limit=200) return 400 VALIDATION_ERROR.

The DB queries use SKIP/TAKE (SQL OFFSET/LIMIT) with a $transaction to
fetch data and total count in one round-trip:

  const [data, total] = await prisma.$transaction([
    prisma.fund.findMany({ skip: (page-1)*limit, take: limit }),
    prisma.fund.count(),
  ]);


================================================================================
  10. WHAT I WOULD MENTION IN AN INTERVIEW
================================================================================

1. "THE DECORATOR PATTERN GIVES YOU CLEAN SRP"
   Each service decorator has one job. If someone asks me to add rate
   limiting or metrics — I add a decorator. Nothing existing changes.
   This is the Open/Closed principle in practice.

2. "RESULT<T,E> MAKES ERROR HANDLING EXPLICIT IN THE TYPE SYSTEM"
   TypeScript can't type thrown exceptions. Result<T,E> makes failure
   cases part of the function signature. The caller is forced to handle
   both branches. It's similar to Rust's Result or C#'s FluentResults.

   What is a Result monad?
A monad is just a fancy word for a wrapper type that carries either a success value or a failure value — and forces you to check which one you got before using it. Think of it as a box that can hold one of two things: a result, or an error.

In this codebase it's defined in src/lib/result.ts:


Result<T, E>
// T = the success type  (e.g. FundResponseDto)
// E = the error type    (e.g. DomainError)
The box can be created in two ways:


Result.ok(fund)           // success — wraps the value
Result.fail(new FundNotFoundError(id))  // failure — wraps the error
The problem it solves
Without Result — the old way:


// Service throws
async findById(id: string): Promise<FundResponseDto> {
  const fund = await this.fundRepo.findById(id);
  if (!fund) throw new Error('Not found');  // 💥 thrown somewhere deep
  return fund;
}

// Controller must catch — and guess what might have been thrown
try {
  const fund = await service.findById(id);
  res.json(fund);
} catch (err) {
  // What kind of error? 404? 500? We don't know statically.
  res.status(500).json({ error: 'Something went wrong' });
}
With Result — this project's way:


// Service returns, never throws
async findById(id: string): Promise<Result<FundResponseDto, DomainError>> {
  const fund = await this.fundRepo.findById(id);
  if (!fund) return Result.fail(new FundNotFoundError(id));  // value, not exception
  return Result.ok(toFundResponse(fund));
}
See fund.service.ts:40-48.

How the controller uses it
In fund.controller.ts:23-28:


const result = await this.service.findById(id);

if (result.isErr) {
  return res.status(result.error.statusCode)  // 404, 409, 500 — already on the error object
             .json({ error: { code: result.error.code, message: result.error.message } });
}

return res.json(result.value);  // safe to access — we checked isErr first
The controller doesn't need to know what might throw or catch exceptions — it just checks isErr and the error already carries its HTTP status code.

The error hierarchy
Defined in src/errors/domain-errors.ts:


DomainError (abstract)
├── NotFoundError         → statusCode 404, code "NOT_FOUND"
│   ├── FundNotFoundError
│   └── InvestorNotFoundError
├── ValidationError       → statusCode 400, code "VALIDATION_ERROR"
├── DuplicateEmailError   → statusCode 409, code "CONFLICT"
└── InfrastructureError   → statusCode 500, code "INTERNAL_ERROR"
Each error knows its own HTTP status, so the controller just reads result.error.statusCode — no switch or instanceof needed.

Summary
Concept	What it means here
Result<T, E>	A box: either holds a value T or an error E
Result.ok(x)	"it worked, here's the data"
Result.fail(e)	"it failed, here's why"
result.isErr	check before unwrapping
result.value	the success data (safe after isErr check)
result.error	the domain error with .statusCode and .code built in
The key benefit: errors become plain values you pass around. No surprises, no hidden throws, and the HTTP status mapping is automatic because the error object carries it.

3. "THE BUG I CAUGHT: update() PASSING id IN DATA"
   The original service did:
     const { id, ...updateData } = data;
     await repo.update(id, data);  // BUG: should be updateData
   The fix was to make the repo interface Omit<UpdateFundDto, 'id'> —
   TypeScript now makes this a compile error. Type safety caught a runtime bug.

4. "CACHE INVALIDATION IS THE HARD PART"
   When a fund is created/updated, I invalidate by PREFIX not exact key
   because there could be many cached pages (page=1, page=2, etc.).
   The prefix and key builder are derived from the same constant so they
   can never drift apart.

5. "pino-pretty GOTCHA IN PRODUCTION"
   pino-pretty is a dev-only pretty-printer. If you try to use it without
   installing it, pino throws at startup. I guard it with an env flag
   (LOG_PRETTY=true) so it's only activated when explicitly opted in.
   Discovered this when tests crashed — good learning moment.

6. "SINGLE-FLIGHT FOR CACHE STAMPEDES"
   I mention that lib/single-flight.ts exists and explain why: under
   concurrent load, many requests can simultaneously miss the cache and
   all hit the DB. Single-flight deduplicates: only one DB call per unique
   cache key at a time. All other callers share that result.
   (Built it, decided it's a stretch goal, documented in README.)

7. "LAZY DI RESOLUTION IN ROUTES"
   An interesting debugging moment: app.ts imports routes which import
   the DI container. If routes resolve controllers at module load time,
   configureContainer() hasn't run yet → crash. The fix: resolve lazily
   inside each request handler. The container is only queried AFTER
   configureContainer() has been called at startup.

8. "WHY SYMBOL.FOR() NOT SYMBOL()"
   Symbol() creates a NEW unique symbol every time. Symbol.for('key') returns
   the SAME symbol for the same key globally. If the same token string is
   used in two different modules (can happen with duplicate imports or
   module reloading in tests), Symbol.for ensures they're the same token.

  FINAL ARCHITECTURE SUMMARY
================================================================================

HTTP Request
    │
    ▼
Express Router + Middleware (Zod validation, error handler)
    │
    ▼
Controller (parse req → call service → send res)
    │ @inject(TOKENS.FundService)
    ▼
CachingFundService (decorator)
    │  cache hit → return immediately
    │  cache miss → call inner
    ▼
LoggingFundService (decorator)
    │  delegate to inner, log result + timing
    ▼
FundService (core — pure business logic)
    │ @inject(TOKENS.FundRepo)
    ▼
PrismaFundRepository (implements IFundRepository)
    │
    ▼
PrismaClient → PostgreSQL

All registered in container.ts.
